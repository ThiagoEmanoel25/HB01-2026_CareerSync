Review da implementação de auth email/senha + JWT

Relatório técnico sobre a feature implementada em 31/05/2026: autenticação própria com JWT, isolamento de análises por usuário, rotas frontend com analysisId e histórico vindo do servidor.

Status geral: bom desenho, com ajustes antes de merge Frontend build: passou Pytest: não executou por ambiente incompleto

Resumo executivo

Correto

Arquitetura principal

A escolha por módulo isolado de auth foi seguida: segurança em core/security.py, rotas em routers/auth.py, schemas próprios e dependência get_current_user.

Correto

Dados por usuário

Analysis.user_id aponta para users.id, novas análises gravam o dono e as buscas por id retornam 404 quando a análise não pertence ao usuário logado.

Ajustar

Estado local pós-auth

O ponto mais sensível está no frontend: o estado persistido da análise não é limpo no logout e o AnalysisGate confia no Zustand para liberar uma análise sem validar no servidor.

O que foi implementado

Backend
  • config.py: jwt_secret obrigatório e jwt_expire_days = 7.
  • backend/.env.example: adiciona JWT_SECRET=.
  • requirements.txt: adiciona passlib, bcrypt==4.0.1, python-jose[cryptography] e email-validator.
  • db_models.py: cria User com __tablename__ = "users" e adiciona Analysis.user_id.
  • database.py: inclui user_id em _ANALYSIS_COLUMNS_TO_ENSURE.
  • core/security.py: hash bcrypt, verificação de senha, JWT HS256 e dependency de usuário atual.
  • routers/auth.py: registro, login e /auth/me.
  • routers/analysis.py: proteção em todas as rotas, checagem de posse e novo GET /analysis.
  • context.py e interview.py: rotas passam a exigir autenticação.
Frontend
  • store/auth.ts: store persistido com token, usuário e logout.
  • lib/api.ts: injeta Bearer, trata 401 com token e adiciona hooks de auth/listagem.
  • pages/Login e pages/Register: telas públicas com redirect para /new.
  • router.tsx: RequireAuth, rotas públicas e rotas /analysis/:analysisId/*.
  • AnalysisGate: carrega detalhe + summary por id da URL e hidrata useSession.
  • HistoryList: usa histórico server-side.
  • Sidebar: mostra email e botão sair.
  • NavigationMenu e páginas internas: navegação agora usa rotas com id da análise.

Pontos fortes

Isolamento de ownership no backend

A função _get_analysis_or_404 centraliza a validação de posse. Isso reduz o risco de esquecer filtro por usuário nas sub-rotas de análise.

URLs persistentes por análise

A troca para /analysis/:analysisId/* foi a decisão certa. O histórico server-side passa a funcionar com refresh, link direto e navegação do browser.

Tratamento de credenciais

Registro normaliza email; login usa mensagem genérica; senha curta fica a cargo do Pydantic; bcrypt foi pinado para evitar incompatibilidade conhecida com passlib.

Escopo de segurança simples

Access token único de 7 dias, sem refresh token e sem blacklist, é coerente com o escopo do produto e reduz complexidade operacional.

Críticas e riscos

P1

Logout não limpa o estado persistido da análise

Sidebar.handleLogout() chama apenas useAuth.logout(). O store prep-ai-session continua com analysisId, matchScore, summary, gaps e outros dados da conta anterior. Depois de login com outro usuário no mesmo browser, o menu pode continuar habilitado e o AnalysisGate pode liberar a tela sem validar no backend se o id local bater.

Referências: frontend/src/components/Sidebar/index.tsx:24, frontend/src/components/AnalysisGate/index.tsx:26.

Correção recomendada: no logout, limpar useSession, useProgress se necessário e cache do React Query. Melhor ainda: guardar no session store o userId dono do snapshot e considerar carregado apenas se esse dono bater com auth.user.id.

P1

AnalysisGate confia demais no Zustand

A linha alreadyLoaded = sessionAnalysisId === analysisId && matchScore !== null faz o gate pular completamente as requests de detalhe/summary. Isso é útil para performance, mas vira bypass de validação client-side quando há estado persistido antigo ou troca de conta.

Referência: frontend/src/components/AnalysisGate/index.tsx:26-30.

Correção recomendada: sempre validar ownership pelo servidor ao entrar numa URL de análise, ou condicionar o fast-path a um snapshot marcado como criado/carregado para o usuário autenticado atual.

P2

Renderiza filhos antes de hidratar o store

Quando detail.data e summary.data chegam, o gate retorna <Outlet /> antes do useEffect aplicar os dados no Zustand. Isso pode causar um frame com dados vazios ou da análise anterior.

Referências: frontend/src/components/AnalysisGate/index.tsx:32-38, frontend/src/components/AnalysisGate/index.tsx:63.

Correção recomendada: hidratar de forma síncrona antes de renderizar o Outlet, ou trocar as páginas para lerem direto do React Query em vez de dependerem do Zustand como intermediário.

P2

Requests de TTS não passam pelo cliente HTTP central

useInterviewTTS monta fetch manualmente. Ele envia o Bearer, mas não usa apiRequest; por isso, um 401 de token expirado não dispara o logout/redirect centralizado.

Referência: frontend/src/lib/api.ts:449-469.

Correção recomendada: criar uma variante de apiRequest para respostas binárias/stream ou reutilizar a mesma rotina de tratamento de 401 no fetch manual.

P2

Testes antigos ainda parecem desatualizados para auth

O novo test_auth_router.py cobre registro, login, ownership e rotas protegidas. Porém test_analysis_router.py e test_code_challenges.py ainda fazem chamadas /analysis/* sem header de autorização em vários pontos. Com auth obrigatória, esses testes devem ser atualizados ou vão falhar quando a suíte rodar de fato.

Correção recomendada: criar helper de usuário/token nos testes legados e criar análises via rota autenticada ou inserir user_id compatível no setup.

P3

URL do currículo agora exige header, mas é retornada como link simples

AnalysisDetailResponse.resume.url continua retornando /analysis/{id}/resume. Como a rota agora exige Bearer, abrir esse URL diretamente no browser não envia o token. Se o app usar esse link para preview/download sem fetch autenticado, o PDF vai falhar com 401.

Correção recomendada: se houver preview/download, buscar o PDF via fetch autenticado e criar ObjectURL, ou adicionar um endpoint com URL assinada temporária.

P3

Header Bearer é tratado de forma case-sensitive

get_current_user exige Authorization começando exatamente por Bearer . O frontend atual envia assim, então funciona, mas o esquema HTTP é normalmente tratado como case-insensitive.

Correção recomendada: aceitar bearer em qualquer caixa usando split e comparação com .lower().
